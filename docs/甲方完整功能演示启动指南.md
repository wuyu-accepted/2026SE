# 甲方完整功能演示启动指南

本文档用于在演示机器上启动“学院学生综合服务与党团管理平台”的完整功能环境，并明确每一步启动了什么、如何判断启动成功、演示时应走哪些功能路径。

## 1. 可行性结论

推荐路线可行：使用 `deploy/knowledge` 目录下的一键 Docker 编排启动完整演示环境。

该路线会启动三类服务：

| 服务 | 容器名 | 作用 | 默认访问 |
| --- | --- | --- | --- |
| PostgreSQL | `2026se-knowledge-db` | 业务数据库，保存账号、通知、知识库、审批等数据 | `127.0.0.1:5432` |
| Spring Boot 后端 | `2026se-knowledge-backend` | 统一 API、鉴权、业务逻辑、文件解析、OCR、索引、语义检索 | `http://127.0.0.1:18080/api` |
| Vue 管理端 | `2026se-knowledge-frontend` | 管理端页面，Nginx 托管前端并代理 `/api` 到后端 | `http://127.0.0.1:8080` |

本路线比单独启动 H2 后端更适合甲方演示，因为它包含前端、后端、数据库和知识库智能能力，并且数据保存在 `runtime/` 目录中，服务重启后不会丢失。

不要把仓库根目录的下面命令当作完整体验启动命令：

```bash
docker compose -f docker-compose.postgres.yml up -d --build
```

该命令只启动普通后端镜像和 PostgreSQL，适合基础后端接口联调；它不启动 Vue 管理端前端，不使用 `demo/Dockerfile.knowledge` 知识库增强镜像，也不完整准备 OCR、全文索引、向量索引和本地模型目录。给甲方体验完整功能时，应使用本指南后续的 `./deploy/knowledge/bootstrap.sh` 和 `./deploy/knowledge/start.sh`。

## 2. 当前工作区核验结果

已核验项目结构：

| 目录/文件 | 说明 |
| --- | --- |
| `demo/` | Spring Boot 后端，入口为 `com.ruc.platform.PlatformApplication` |
| `web-counselor/` | Vue 3 + Vite 管理端 |
| `miniprogram-1/` | 微信小程序学生端 |
| `deploy/knowledge/docker-compose.yml` | 完整演示环境 Docker Compose 编排 |
| `deploy/knowledge/*.sh` | 初始化、启动、状态、日志、停止脚本 |
| `runtime/` | 数据库、上传文件、索引和模型持久化目录 |

已核验 Compose 配置：

- `db` 使用 `postgres:16`，带 `pg_isready` 健康检查。
- `backend` 使用 `demo/Dockerfile.knowledge` 构建，等待数据库 healthy 后启动。
- `backend` 健康检查为 `curl -fsS http://127.0.0.1:18080/api/health`。
- `frontend` 使用 `web-counselor/Dockerfile` 构建，构建参数 `VITE_API_BASE_URL=/api`。
- 前端 Nginx 配置会把 `/api/` 代理到 `http://backend:18080/api/`。

当前机器发现两个演示前必须处理的问题：

| 问题 | 当前现象 | 影响 | 处理方式 |
| --- | --- | --- | --- |
| Docker 权限不足 | `docker info` 报 `permission denied while trying to connect to the docker API` | 当前用户无法启动容器 | 使用有 Docker 权限的账号，或将当前用户加入 `docker` 组后重新登录 |
| `runtime/` 权限异常 | `runtime` 当前归属为 `nobody:nogroup` | `bootstrap.sh` 会因当前用户不可写而失败 | 执行 `sudo chown -R $(id -u):$(id -g) runtime` |

只要处理以上两点，启动路线本身是成立的。

## 3. 演示机器准备

演示机器建议使用 Linux 服务器、Linux 桌面机、macOS 或 Windows WSL。

必备条件：

- 已安装 Docker。
- Docker 服务已启动。
- Docker Compose V2 可用，即 `docker compose version` 能正常输出。
- 当前用户可执行 Docker 命令。
- 首次构建时能访问 Docker 镜像源、Maven 源、npm 源和 Debian/Ubuntu 软件源。
- 默认端口 `8080`、`18080`、`5432` 未被占用。

演示前检查：

```bash
cd /home/octopus/2026SE
docker compose version
docker info
```

`docker info` 必须能输出 Server 信息。如果报权限错误，先修复 Docker 权限。

如果当前工作区已有 `runtime/` 权限问题，先执行：

```bash
sudo chown -R $(id -u):$(id -g) runtime
```

## 4. 初始化

在仓库根目录执行：

```bash
cd /home/octopus/2026SE
./deploy/knowledge/bootstrap.sh
```

这一步会做以下事情：

- 检查 Docker 是否安装并可访问。
- 检查 Docker Compose 是否可用。
- 如果 `deploy/knowledge/.env` 不存在，则从 `.env.example` 生成。
- 创建或检查 `runtime/` 持久化目录。

初始化完成后，可查看配置文件：

```bash
sed -n '1,120p' deploy/knowledge/.env
```

重点确认：

```text
FRONTEND_PORT=8080
BACKEND_PORT=18080
DB_PORT=5432
SPRING_PROFILES_ACTIVE=postgres
KNOWLEDGE_OCR_ENABLED=true
KNOWLEDGE_SEMANTIC_ENABLED=true
```

如端口冲突，可编辑 `deploy/knowledge/.env`：

```text
FRONTEND_PORT=8081
BACKEND_PORT=18081
DB_PORT=15432
```

## 5. 启动完整环境

执行：

```bash
./deploy/knowledge/start.sh
```

该命令会执行：

- 构建后端知识库增强镜像。
- 构建前端 Nginx 镜像。
- 启动 PostgreSQL。
- 等数据库健康后启动后端。
- 启动前端。

启动后脚本会打印访问地址，例如：

```text
管理端前端：http://localhost:8080
后端接口：http://localhost:18080/api
```

如果部署在服务器，把 `localhost` 换成服务器 IP 或域名。

如果不想使用脚本，可以在仓库根目录手动执行等价命令：

```bash
cp deploy/knowledge/.env.example deploy/knowledge/.env

mkdir -p \
  runtime/postgres \
  runtime/uploads \
  runtime/lucene/knowledge \
  runtime/lucene/knowledge-vectors \
  runtime/models/embedding \
  runtime/logs

docker compose \
  --env-file deploy/knowledge/.env \
  -f deploy/knowledge/docker-compose.yml \
  up -d --build
```

手动方式和脚本方式启动的是同一套 `db`、`backend`、`frontend` 服务。脚本只负责前置检查和提示，不是后端功能依赖。

## 6. 科学验收步骤

### 6.1 查看容器状态

```bash
./deploy/knowledge/status.sh
```

应看到类似服务：

```text
2026se-knowledge-db
2026se-knowledge-backend
2026se-knowledge-frontend
```

判断标准：

- `db` 状态为 `Up`，最好显示 `healthy`。
- `backend` 状态为 `Up`。
- `frontend` 状态为 `Up`。

### 6.2 验证后端健康接口

```bash
curl http://127.0.0.1:18080/api/health
```

正常返回应包含：

```json
{
  "code": 0,
  "data": {
    "status": "UP"
  }
}
```

### 6.3 验证前端页面

浏览器访问：

```text
http://127.0.0.1:8080
```

能打开登录页即说明前端 Nginx 已启动。

### 6.4 验证前后端联通

在管理端登录：

| 角色 | 账号 | 密码 |
| --- | --- | --- |
| 超级管理员 | `admin` | `admin123` |
| 辅导员 | `counselor` | `counselor123` |

能登录并进入后台，说明：

- 前端构建正常。
- Nginx `/api` 代理正常。
- 后端接口正常。
- 数据库迁移和初始化数据正常。

## 7. 甲方演示路径

建议按“管理端能力 -> 业务闭环 -> 学生端体验”的顺序演示。

### 7.1 管理端登录

访问：

```text
http://演示机器IP:8080
```

先用管理员账号登录：

```text
admin / admin123
```

可展示：

- 后台整体布局。
- 学生管理。
- 通知发布与管理。
- 知识库文章和模板维护。
- 知识库分类、同义词、推荐权重、索引能力。
- 党团管理。
- 系统设置与 AI 助手配置。

再用辅导员账号登录：

```text
counselor / counselor123
```

可展示：

- 待处理审批。
- 已处理审批。
- 学生信息查询。
- 通知和反馈处理。
- 党团流程相关管理。

### 7.2 知识库智能能力

建议演示：

- 新建或编辑知识条目。
- 上传 PDF、Word、Excel、PPT、图片、Markdown、HTML、CSV 或 ZIP 附件。
- 发布知识内容。
- 搜索关键词，观察标题、正文、分类筛选和高亮。
- 展示同义词、搜索建议、相似结果或语义检索效果。
- 如果上传图片或扫描 PDF，说明 OCR 会异步处理。

注意：

- 未放置 ONNX 模型时，语义检索会使用本地 hash embedding fallback，功能能跑通，但语义质量不等同于正式中文向量模型。
- 如需真实本地中文向量模型，把 `model.onnx` 和 `vocab.txt` 放到：

```text
runtime/models/embedding/model.onnx
runtime/models/embedding/vocab.txt
```

### 7.3 通知与反馈

建议演示：

- 管理端发布通知。
- 设置通知范围、状态和内容。
- 学生端查看通知。
- 学生端提交反馈。
- 管理端查看并处理反馈。

### 7.4 请假审批闭环

建议演示：

- 辅导员管理端查看待审批申请。

## 8. 日志与排障

查看全部日志：

```bash
./deploy/knowledge/logs.sh
```

只看后端日志：

```bash
./deploy/knowledge/logs.sh backend
```

只看数据库日志：

```bash
./deploy/knowledge/logs.sh db
```

只看前端日志：

```bash
./deploy/knowledge/logs.sh frontend
```

常见问题：

| 问题 | 判断方式 | 处理方式 |
| --- | --- | --- |
| Docker 权限不足 | `docker info` 报 socket permission denied | 使用有权限账号，或加入 docker 组后重新登录 |
| `runtime/` 不可写 | `bootstrap.sh` 提示 runtime 当前用户不可写 | `sudo chown -R $(id -u):$(id -g) runtime` |
| 端口冲突 | `start.sh` 或 Compose 报端口已占用 | 修改 `deploy/knowledge/.env` 中的端口 |
| 后端未启动 | `curl /api/health` 失败 | 看 `./deploy/knowledge/logs.sh backend` |
| 数据库未就绪 | backend 等待或连接失败 | 看 `./deploy/knowledge/logs.sh db` |
| OCR 构建失败 | 镜像构建时 apt 下载失败 | 配置系统软件源，或临时设 `KNOWLEDGE_OCR_ENABLED=false` |
| 小程序请求失败 | 小程序接口报网络错误 | 确认 `BASE_URL` 为演示机器 IP，关闭域名校验 |

## 9. 停止服务

演示结束后执行：

```bash
./deploy/knowledge/stop.sh
```

该命令会停止并移除容器，但不会删除 `runtime/` 中的数据。

下次演示继续执行：

```bash
./deploy/knowledge/start.sh
```

## 10. 数据持久化与清理

演示数据保存在：

```text
runtime/postgres
runtime/uploads
runtime/lucene
runtime/models
```

如果要保留演示环境，不要删除 `runtime/`。

如果要重置演示数据，需要先停止服务，再清理对应目录。清理数据库属于破坏性操作，执行前应确认已经备份。

## 11. 最终建议

正式给甲方演示前，建议至少提前半天完成以下预演：

1. 修复 Docker 权限和 `runtime/` 权限。
2. 执行 `bootstrap.sh` 和 `start.sh`。
3. 确认 `status.sh` 中三个服务均为 `Up`。
4. 确认 `/api/health` 返回 `UP`。
5. 用管理员、辅导员、学生账号各登录一次。
6. 准备 1 个知识库上传文件、1 条通知、1 条请假申请作为演示素材。
7. 若使用服务器演示，提前把小程序 `BASE_URL` 改为服务器 IP。

完成以上检查后，这条启动路线可以支撑完整功能展示。
