# 培养方案课程 JSON 生成工作流

## 目标

通过 OCR 和大模型解析 PDF 培养方案文件，提取所有专业中以下三类课程的信息：

- 部类核心课
- 部类基础课
- 专业核心课

最终生成一个稳定、可校验、可被后端学业分析模块直接读取的课程映射 JSON 文件。该 JSON 用于匹配学生导入的已修读课程，计算已修读学分、未修课程、按类别完成情况，并在小程序学业分析页面展示。

## 目标 JSON 内容

每个专业下，需要保存课程名称、课程类别、学分、开设学期等信息。

建议 JSON 结构如下：

```json
{
  "version": "2024",
  "source": {
    "fileName": "2024级理工大类培养方案-信息学院.pdf",
    "generatedAt": "2026-05-27",
    "method": "ocr+llm"
  },
  "majors": [
    {
      "majorName": "计算机科学与技术",
      "categories": [
        {
          "categoryName": "部类核心课",
          "courses": [
            {
              "courseName": "程序设计",
              "credits": 3,
              "terms": ["第一学期"],
              "rawTerm": "1",
              "required": true
            }
          ]
        },
        {
          "categoryName": "部类基础课",
          "courses": []
        },
        {
          "categoryName": "专业核心课",
          "courses": []
        }
      ]
    }
  ]
}
```

字段说明：

- `version`：培养方案适用年级或版本，例如 `2024`。
- `source.fileName`：原始 PDF 文件名。
- `majorName`：专业名称，必须与系统中学生画像/专业字段尽量一致。
- `categoryName`：只允许出现 `部类核心课`、`部类基础课`、`专业核心课` 三类。
- `courseName`：课程名称，去掉多余空格、换行、脚注符号。
- `credits`：学分，使用数字类型。
- `terms`：标准化后的开设学期，可为多个学期。
- `rawTerm`：PDF 原表格中的原始学期字段，便于人工复核。
- `required`：这三类课程默认视为必修或核心要求，先统一设为 `true`。

## 一步步完成流程

### 第一步：收集并规范原始 PDF

1. 收集所有需要支持的培养方案 PDF。
2. 按年级、学院、专业范围命名文件，例如：

```text
2024级理工大类培养方案-信息学院.pdf
2024级人文社科大类培养方案-xxx学院.pdf
```

3. 将 PDF 放入后端可访问的固定目录，例如：

```text
demo/src/main/resources/study-plan/
```

4. 记录每个 PDF 的适用范围：年级、学院、专业。

### 第二步：PDF 转图片或文本

根据 PDF 类型选择处理方式：

1. 如果 PDF 是可复制文本型，优先直接抽取文本和表格。
2. 如果 PDF 是扫描件或表格识别效果差，先将每页转为图片，再进行 OCR。
3. 每页 OCR 后保留页码，输出中间结果：

```text
pageNumber
ocrText
```

4. 不要在这一步直接丢弃页码，因为后续人工复核需要知道课程来自 PDF 哪一页。

### 第三步：按专业切分内容

1. 从 OCR 文本中识别专业标题，例如：

```text
计算机科学与技术专业
软件工程专业
信息安全专业
数据科学与大数据技术专业
```

2. 将每个专业对应的培养方案内容切成独立片段。
3. 每个片段至少包含：

```json
{
  "majorName": "专业名称",
  "pages": [12, 13, 14],
  "text": "该专业对应的 OCR 文本"
}
```

4. 如果一个专业跨多页，需要合并连续页内容，不能只取标题所在页。

### 第四步：定位目标课程类别

对每个专业片段，只提取以下三个类别：

```text
部类核心课
部类基础课
专业核心课
```

处理规则：

1. 找到类别标题后，读取其下方课程表。
2. 直到遇到下一个课程类别标题、下一个专业标题或表格结束为止。
3. 忽略不在目标范围内的类别，例如通识课、公共课、个性化选修课、实践教学、毕业论文等。
4. 如果 OCR 将类别标题识别错，例如 `部类棱心课`，大模型可以根据上下文推断，但必须在结果中保留 `warnings` 供人工复核。

### 第五步：大模型抽取课程信息

对每个专业片段调用大模型，要求模型只输出 JSON，不输出解释性文字。

建议提示词如下：

```text
你是高校培养方案结构化抽取助手。

任务：
从下面的培养方案 OCR 文本中，提取指定专业的三类课程信息：
1. 部类核心课
2. 部类基础课
3. 专业核心课

只提取课程名称、学分、开设学期。
不要提取通识课、公共课、选修课、实践教学、毕业论文、军事训练、创新创业等非目标类别。

输出要求：
1. 必须输出合法 JSON。
2. 不要输出 Markdown。
3. categoryName 只能是：部类核心课、部类基础课、专业核心课。
4. credits 必须是数字。
5. terms 使用标准中文学期，例如：第一学期、第二学期、第三学期。
6. 如果原文中学期是数字 1-8，请转换为 第一学期 到 第八学期，并把原始值放入 rawTerm。
7. 如果课程跨多个学期，请 terms 输出数组。
8. 如果某个字段不确定，放入 warnings，不要编造。
9. 保留 sourcePages，便于人工复核。

输出 JSON 格式：
{
  "majorName": "",
  "sourcePages": [],
  "categories": [
    {
      "categoryName": "部类核心课",
      "courses": [
        {
          "courseName": "",
          "credits": 0,
          "terms": [],
          "rawTerm": "",
          "sourceText": "",
          "warnings": []
        }
      ]
    }
  ],
  "warnings": []
}

专业名称：
{{majorName}}

OCR 文本：
{{ocrText}}
```

### 第六步：清洗和标准化

大模型输出后，后端脚本需要做二次清洗，不能直接入库或直接作为最终 JSON。

清洗规则：

1. 课程名称：
   - 去除换行、多余空格。
   - 去除脚注符号，如 `*`、`①`、`注1`。
   - 保留罗马数字和课程序号，如 `高等数学Ⅰ`、`大学英语Ⅱ`。

2. 学分：
   - 转成数字。
   - 支持 `3`、`3.0`、`2.5`。
   - 如果无法识别，放入 `validationErrors`。

3. 学期：
   - `1` 转成 `第一学期`。
   - `2` 转成 `第二学期`。
   - `1/2`、`1,2` 转成 `["第一学期", "第二学期"]`。
   - `秋季`、`春季` 如无法对应具体学期，应保留原文并加入警告。

4. 类别：
   - 只保留三类目标课程。
   - 类别名必须标准化为固定中文。

5. 去重：
   - 同一专业、同一类别、同一课程名重复时合并。
   - 如果学分或学期冲突，保留多来源并加入 `validationErrors`。

### 第七步：人工复核

生成初版 JSON 后，需要输出复核清单：

```json
{
  "missingCredits": [],
  "missingTerms": [],
  "conflicts": [],
  "lowConfidenceItems": [],
  "unknownCategories": []
}
```

人工重点检查：

1. 每个专业是否都被识别。
2. 三个目标类别是否完整。
3. 课程名称是否被 OCR 拆断。
4. 学分是否错位到相邻课程。
5. 开设学期是否被识别成学时、序号或页码。
6. 不应出现非目标类别课程。

复核完成后，再生成最终 JSON。

### 第八步：后端落地文件

建议将最终 JSON 放到后端资源目录：

```text
demo/src/main/resources/study-plan/course-catalog.json
```

后端提供一个课程目录加载类，启动时读取该 JSON：

```text
StudyPlanCourseCatalog
```

加载后在内存中建立索引：

```text
majorName -> categoryName -> courseName -> courseInfo
normalizedCourseName -> courseInfo
```

其中 `normalizedCourseName` 用于匹配学生导入的课程名，规则包括：

1. 去空格。
2. 全角半角归一。
3. 中文括号和英文括号归一。
4. 罗马数字保留但做兼容映射，例如 `I` 和 `Ⅰ`。

### 第九步：对接学业分析

学生导入已修读课程后，学业分析流程如下：

1. 小程序端导入或录入已修读课程。
2. 后端保存学生课程记录。
3. 后端根据学生专业读取对应专业的课程目录。
4. 对每门已修课程做课程名标准化匹配。
5. 匹配成功后读取课程学分和类别。
6. 按类别计算：
   - 已修课程数
   - 已修学分
   - 应修总学分
   - 未修课程
   - 完成比例
7. 返回给小程序展示。

### 第十步：页面展示建议

小程序学业分析页面建议展示：

1. 总体完成情况。
2. 三类课程分别的完成进度：
   - 部类核心课
   - 部类基础课
   - 专业核心课
3. 已匹配课程列表。
4. 未匹配课程列表。
5. 未修课程列表。
6. 识别失败提示，例如：

```text
以下课程未在培养方案中匹配到，请检查课程名称是否与培养方案一致。
```

## 后续可拆分任务

1. 编写 PDF OCR 脚本。
2. 编写专业内容切分脚本。
3. 编写大模型抽取脚本。
4. 编写 JSON schema 校验。
5. 编写人工复核报告生成器。
6. 将最终 JSON 放入后端资源目录。
7. 改造后端学业分析服务，读取 JSON 课程目录。
8. 改造小程序学业分析页面，展示三类课程完成情况。

## 验收标准

1. 每个专业都能在 JSON 中找到。
2. 每个专业只包含三类目标课程。
3. 每门课程至少包含课程名称、学分、开设学期。
4. 后端启动时能成功读取 JSON。
5. 学生导入已修读课程后，系统能自动匹配课程并计算学分。
6. 未匹配课程会明确展示，不会静默丢失。
7. 人工抽查 PDF 原文和 JSON 内容，课程名称、学分、学期基本一致。
