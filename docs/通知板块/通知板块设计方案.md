# 通知板块设计方案

## 1. 背景与目标

通知板块负责连接管理端发布能力与学生端消息接收能力，覆盖以下范围：

- 学生端通知列表与消息已读。
- 管理端通知发布、编辑、列表和状态管理。
- 后端通知管理 API、消息投递逻辑和学生筛选发布。
- 基于现有 `notice`、`user_message` 表实现，不优先新增复杂表结构。

当前项目已经具备学生端基础消息列表和已读接口，但管理端通知发布能力尚未落地。因此本方案以“先完成可验收闭环，再逐步增强”为原则设计。

## 2. 当前实现盘点

### 2.1 已实现部分

#### 学生端

- 页面路径：`miniprogram-1/pages/notice/notice`
- 已调用接口：`GET /api/messages/recent?limit=20`
- 已读接口：`POST /api/messages/{id}/read`
- 当前展示方式：通知卡片列表，点击后使用弹窗显示摘要并标记已读。

#### 后端学生侧接口

- `GET /api/messages/recent`
- `GET /api/messages/unread-count`
- `POST /api/messages/{id}/read`

#### 数据表

- `notice`：通知内容表。
- `user_message`：用户消息投递表，记录用户维度的阅读状态。

### 2.2 未实现部分

- 管理端 `/notices` 仍为占位页。
- 管理端没有通知列表、创建、编辑、发布、下架页面。
- 后端没有 `/api/admin/notices` 管理端 CRUD 接口。
- 后端没有按学生筛选条件进行投递的逻辑。
- 学生端没有通知详情页，无法展示完整正文。

## 3. 总体设计原则

1. **复用现有模型**：继续使用 `notice` 和 `user_message`，避免第一阶段引入过多新表。
2. **管理端负责发布，学生端负责接收**：管理端操作 `notice`，发布时投递到 `user_message`。
3. **筛选条件取交集**：年级、专业、班级、身份同时填写时，只有全部匹配的学生会收到通知。
4. **空条件表示不限**：某个筛选字段为空时不参与过滤。
5. **发布动作显式触发**：创建草稿不投递，只有点击“发布”后才生成用户消息。
6. **第一阶段先做闭环**：先完成列表、筛选发布和学生接收；通知详情、撤回同步等作为第二步增强。

## 4. 功能范围

### 4.1 第一步：完成通知发布闭环

第一步目标是让通知板块具备课程验收所需的完整主链路。

### 管理端能力

- 查看通知列表。
- 按标题关键词、通知类型、状态筛选通知。
- 新建通知草稿。
- 编辑通知。
- 按筛选条件发布通知。
- 下架通知。
- 删除草稿或已下架通知。
- 查看每条通知的投递人数。

### 后端能力

- 新增 `/api/admin/notices` 管理端接口。
- 支持通知分页查询。
- 支持通知详情查询。
- 支持创建、编辑、发布、下架、删除。
- 发布时按学生档案字段筛选目标用户。
- 将目标用户写入 `user_message`。
- 返回投递人数，便于管理端确认发布结果。

### 学生端能力

- 继续使用现有通知列表。
- 支持查看通知摘要。
- 支持标记已读。

### 4.2 第二步：增强学生端详情体验

第二步目标是让学生能查看通知正文，而不只是摘要。

### 学生端新增能力

- 新增通知详情页，例如：`miniprogram-1/pages/notice-detail/notice-detail`。
- 通知列表点击后跳转详情页，而不是只弹窗。
- 详情页展示：标题、类型、标签、发布时间、正文、已读状态。
- 进入详情页时自动标记已读。

### 后端新增能力

- 新增学生侧详情接口，建议：`GET /api/messages/{id}`。
- 详情接口需要校验消息归属：只能查看当前登录用户自己的 `user_message`。
- 返回内容可以由 `user_message` 关联 `notice` 得到，包含 `notice.content`。

### 管理端增强能力

- 通知列表增加“预览”按钮。
- 预览弹窗展示完整通知正文和投递条件。

### 4.3 第三步：增强精准发布与运营能力

第三步目标是让通知发布更接近真实管理后台。

可选增强包括：

- 发布前预估目标人数。
- 支持按多个年级、多个专业、多个班级发布。
- 支持指定学生名单发布。
- 支持导入学号名单发布。
- 支持定时发布。
- 支持通知撤回后同步移除或置灰学生端消息。
- 支持通知阅读统计：已读人数、未读人数、已读率。
- 支持通知置顶或紧急提醒。

这些能力不建议第一阶段全部实现，避免影响主链路交付。

## 5. 数据设计

### 5.1 现有 `notice` 表

字段用途如下：

| 字段 | 用途 |
|---|---|
| `id` | 通知 ID |
| `title` | 标题 |
| `summary` | 摘要 |
| `content` | 正文 |
| `notice_type` | 通知类型，如教学、党团、就业、生活、其他 |
| `tag` | 标签，如重要、紧急、普通 |
| `status` | 状态，0 草稿、1 已发布、2 已下架 |
| `publish_time` | 发布时间 |
| `priority` | 优先级，0 普通、1 重要、2 紧急 |
| `created_by` | 创建人用户 ID |
| `target_tags` | 目标人群筛选条件，第一阶段保存 JSON 字符串 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 5.2 现有 `user_message` 表

字段用途如下：

| 字段 | 用途 |
|---|---|
| `id` | 用户消息 ID |
| `user_id` | 接收人用户 ID |
| `notice_id` | 对应通知 ID |
| `title` | 冗余通知标题，方便列表查询 |
| `summary` | 冗余通知摘要，方便列表查询 |
| `read_status` | 阅读状态，0 未读、1 已读 |
| `read_time` | 阅读时间 |
| `created_at` | 投递时间 |

### 5.3 筛选条件设计

第一阶段基于 `student_profile` 已有字段实现筛选发布：

| 筛选字段 | 来源字段 | 说明 |
|---|---|---|
| 年级 | `student_profile.grade` | 如 `2023` |
| 专业 | `student_profile.major` | 如 `软件工程` |
| 班级 | `student_profile.class_name` | 如 `软工 1 班` |
| 身份 | `student_profile.auth_type` | `student` 或 `cadre` |

筛选条件示例：

```json
{
  "grade": "2023",
  "major": "软件工程",
  "className": "",
  "authType": "cadre"
}
```

含义：投递给 2023 级、软件工程专业、所有班级、学生骨干身份的学生。

### 5.4 `target_tags` 存储约定

第一阶段不新增表，使用 `notice.target_tags` 保存筛选条件 JSON：

```json
{
  "grade": "2023",
  "major": "软件工程",
  "className": "软工1班",
  "authType": "student"
}
```

空字符串或 `null` 表示不限。

后续如果需要更强筛选能力，可新增 `notice_target_rule` 表，将筛选条件结构化保存。

### 6.2 管理端接口设计

### 6.2.1 分页查询通知

```http
GET /api/admin/notices?pageNum=1&pageSize=10&keyword=报名&noticeType=党团&status=1
```

返回：

```json
{
  "code": 0,
  "data": {
    "pageNum": 1,
    "pageSize": 10,
    "total": 12,
    "records": [
      {
        "id": 1,
        "title": "党团活动报名通知",
        "summary": "本周五前完成报名",
        "noticeType": "党团",
        "tag": "重要",
        "status": 1,
        "priority": 1,
        "publishTime": "2026-05-20T10:00:00",
        "createdAt": "2026-05-20T09:50:00",
        "deliveredCount": 36,
        "target": {
          "grade": "2023",
          "major": "软件工程",
          "className": "",
          "authType": "cadre"
        }
      }
    ]
  }
}
```

### 6.2.2 查询通知详情

```http
GET /api/admin/notices/{id}
```

返回完整正文、目标筛选条件和投递人数。

### 6.2.3 创建通知草稿

```http
POST /api/admin/notices
```

请求：

```json
{
  "title": "党团活动报名通知",
  "summary": "本周五前完成报名",
  "content": "请符合条件的同学及时提交报名材料。",
  "noticeType": "党团",
  "tag": "重要",
  "priority": 1,
  "target": {
    "grade": "2023",
    "major": "软件工程",
    "className": "",
    "authType": "cadre"
  }
}
```

创建后默认状态为草稿，不投递消息。

### 6.2.4 编辑通知

```http
PUT /api/admin/notices/{id}
```

编辑通知基础内容和目标筛选条件。

建议规则：

- 草稿可以完整编辑。
- 已发布通知允许编辑标题、摘要、正文、标签和优先级，但不重新投递。
- 已发布通知如果要改变目标人群，应作为第二阶段能力处理，避免重复投递或漏投。
- 已下架通知允许编辑后重新发布。

### 6.2.5 发布通知

```http
POST /api/admin/notices/{id}/publish
```

发布逻辑：

1. 查询通知。
2. 校验通知存在且不是已发布状态。
3. 解析 `target_tags` 中的筛选条件。
4. 按 `student_profile` 筛选目标学生。
5. 如果没有匹配学生，返回业务错误。
6. 更新 `notice.status = 1`、`publish_time = now()`。
7. 为每个目标学生插入一条 `user_message`。
8. 返回投递人数。

返回：

```json
{
  "code": 0,
  "data": {
    "noticeId": 1,
    "deliveredCount": 36
  }
}
```

### 6.2.6 下架通知

```http
POST /api/admin/notices/{id}/offline
```

第一阶段下架只更新 `notice.status = 2`，不删除已经投递到学生端的 `user_message`。

第二阶段可增加“学生端隐藏已下架通知”的行为。

### 6.2.7 删除通知

```http
DELETE /api/admin/notices/{id}
```

建议规则：

- 草稿可以删除。
- 已下架可以删除。
- 已发布不允许直接删除，应先下架。
- 第一阶段删除 `notice` 前需要考虑 `user_message` 外键约束，因此更推荐第一阶段只允许删除未投递的草稿。

## 9. 发布筛选逻辑

### 9.1 查询目标学生

后端发布时按以下逻辑筛选：

- 只查询 `t_user.status = 1` 的有效用户。
- 只查询 `student_profile` 存在的用户。
- `student_profile.auth_type` 必须是 `student` 或 `cadre`。
- 如果 `grade` 非空，则 `student_profile.grade = grade`。
- 如果 `major` 非空，则 `student_profile.major = major`。
- 如果 `className` 非空，则 `student_profile.class_name = className`。
- 如果 `authType` 非空，则 `student_profile.auth_type = authType`。

### 9.2 投递策略

第一阶段采用一次性投递策略：

- 发布时把符合条件的学生全部写入 `user_message`。
- 后续学生档案变化不会自动新增或移除已投递消息。
- 已发布通知再次发布时不重复投递。
- 已下架后重新发布时需要谨慎处理重复消息，第一阶段可选择先删除旧投递再重新投递，或禁止已下架通知重新发布。为了简单可靠，第一阶段建议禁止已下架通知重新发布，后续再增强。

### 9.3 重复投递防护

第一阶段建议在服务层做防护：

- 如果通知已经是已发布状态，则不允许再次发布。
- 插入 `user_message` 前可先按 `notice_id` 查询是否已有投递记录。
- 如果已有投递记录，返回业务错误，避免重复消息。

后续可增加数据库唯一索引：

```sql
CREATE UNIQUE INDEX uk_user_message_notice_user ON user_message(notice_id, user_id);
```

第一阶段如不新增迁移，可先在服务层控制。

## 10. 权限与安全边界

虽然安全治理计划放在后续统一处理，但通知板块仍需遵守基本边界：

- `/api/messages/**` 仅学生端用户访问自己的消息。
- `/api/admin/notices/**` 仅辅导员或管理员访问。
- 发布通知时 `created_by` 记录当前登录管理端用户 ID。
- 学生消息已读接口必须校验消息归属，避免用户标记他人消息。

当前已有 `markAsRead` 只按 messageId 更新，第二步或第一阶段顺手修复时应改为按 `messageId + userId` 更新。

## 11. 错误处理

后端常见错误：

| 场景 | 建议错误信息 |
|---|---|
| 通知不存在 | 通知不存在 |
| 标题为空 | 通知标题不能为空 |
| 正文为空 | 通知正文不能为空 |
| 状态不允许发布 | 当前状态不允许发布 |
| 未匹配到学生 | 当前筛选条件下没有可投递学生 |
| 重复发布 | 通知已发布，不能重复发布 |
| 删除已发布通知 | 已发布通知请先下架后再删除 |
| 学生访问他人消息 | 消息不存在或无权访问 |

前端展示：

- 使用 `ElMessage.error()` 展示后端错误。
- 发布成功后展示 `已发布，投递 X 人`。
- 删除、下架等危险操作需要二次确认。

## 14. 风险与取舍

### 14.1 使用 `target_tags` 保存 JSON 的取舍

优点：

- 不需要新增表。
- 第一阶段实现快。
- 与现有字段兼容。

缺点：

- 不方便复杂查询。
- 后续统计目标规则时需要解析 JSON。
- 多条件、多选、人群组合能力较弱。

结论：第一阶段可接受；后续复杂化时再拆表。

### 14.2 发布后编辑目标范围的取舍

第一阶段不建议允许已发布通知修改目标范围并重新投递，因为会带来重复消息、漏投、撤回等复杂问题。

建议规则：

- 已发布通知可编辑内容，不改变已投递人群。
- 如果目标人群错了，先下架，再新建一条通知重新发布。

### 14.3 下架后学生端是否还能看到

第一阶段：学生端已收到的消息继续可见。

第二阶段：可在学生详情接口或列表接口中关联 `notice.status`，对已下架通知做隐藏或展示“已下架”。

## 15. 推荐验收路径

1. 管理端登录 `counselor/counselor123`。
2. 打开 `/notices`。
3. 新建通知，筛选条件设置为某个已存在年级或身份。
4. 点击发布，确认提示投递人数大于 0。
5. 小程序用命中的学生账号登录。
6. 进入通知页，确认能看到新通知。
7. 点击通知，确认已读状态变化。
8. 使用不命中筛选条件的学生账号登录，确认看不到该通知。
9. 管理端下架通知，确认列表状态变为已下架。
